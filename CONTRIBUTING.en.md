# Contributing to CUMT Alethic Insight Lab

Thank you for contributing to CUMT Alethic Insight Lab.

## Rule Inheritance

Unless otherwise specified by a repository, all repositories under CUMT Alethic Insight Lab follow the organization-level contribution guidelines and templates defined in the `.github` repository.

If a repository has its own README, CONTRIBUTING.md, Issue templates, Pull Request templates, or workflow rules, the repository-specific rules take precedence.

## Workflow

For non-trivial changes:

1. Search existing Issues and Pull Requests.
2. Open an Issue or join an existing Issue.
3. Discuss scope and approach.
4. Implement in a branch or fork.
5. Submit a Pull Request.
6. Respond to review.
7. Merge only after approval.

## Pull Request Expectations

Pull Requests should be:

- small;
- focused;
- reviewable;
- linked to Issues when applicable;
- tested;
- documented when needed.

Avoid:

- huge drive-by rewrites;
- unrelated formatting changes;
- generated junk files;
- secrets, tokens, credentials, or `.env` files;
- changing project direction without discussion.

## AI Assistance Policy

AI-assisted contributions are allowed.

Contributors must disclose, verify, test, and explain AI-assisted work. Every AI-assisted Pull Request should state:

- model/tool used;
- assisted parts;
- cross-checking method;
- tests actually run;
- contributor understanding of the final solution.

AI output must not be submitted blindly. Contributors must understand, verify, and be able to explain their submissions.

## Security

Never commit API keys, GitHub tokens, Hugging Face tokens, cloud credentials, email credentials, recovery codes, private datasets, model checkpoints, large generated artifacts, `.env` files, or SSH private keys.

Security-sensitive reports should be sent to github@aletheion.cn instead of public Issues.
